# Phase 29.1: サイドバー表示時のターミナル幅補償

## 背景

Cmd+T でタブを追加するとサイドバーが出現し、ターミナル領域の幅がサイドバー分だけ縮小する。
Terminal size（`tput cols`）が減り、既存のテキストレイアウトが崩れる。

## 選択肢

### A: ウィンドウ幅をサイドバー分拡張（短期修正）

サイドバーが出現するタイミングでウィンドウ幅を `sidebar_width_px` 分だけ広げる。
サイドバーが消える（タブが1つに戻る）タイミングで元に戻す。

**利点:** 実装がシンプル。Terminal size が維持される。
**欠点:** ウィンドウが急に大きくなるのは UX として微妙。画面端だとはみ出す。

### B: macOS ネイティブタブ UI に再構成（長期計画）

macOS の `NSWindow` タブ機能（`window.addTabbedWindow`）を使い、サイドバーを廃止する。

```
現在: [サイドバー | ターミナル]  ← 1ウィンドウ内にサイドバー描画
将来: [タブバー]                 ← macOS ネイティブのタブバー
      [ターミナル]               ← 全幅使用
```

**利点:**
- ターミナル幅が常に最大（サイドバー分のロスなし）
- Chrome-like UX（ドラッグでタブの切り出し/結合）が OS レベルでサポートされる
- macOS の「すべてのウィンドウを結合」メニューが使える
- SDI ファーストの哲学に完全に合致（タブが1つのときはタブバー非表示）

**欠点:**
- 大掛かりなリファクタリング（Session/Surface 分離の本格実装が必要）
- winit が `NSWindow` タブを直接サポートしていないため、`objc2` で直接操作が必要
- 他 OS 対応時にサイドバー実装も残す必要がある

### C: リサイズインクリメント方式

サイドバー出現時にターミナル幅を維持するためリサイズするが、セルの整数倍を保つ。
ウィンドウ幅 = (現在のカラム数 + スクロールバー) * cell_width + sidebar_width + padding。

## 推奨

短期: **A** または **C** でターミナル幅の維持を実装。
長期: **B** の macOS ネイティブタブへの移行を計画。ただし他 OS でのサイドバー実装は DI 設計で残す。
