# CJK 文字レンダリング検証テクニック

## 対照群ツール: render-text

`tools/test-utils/render-text` は CoreText を使ってテキストを PNG にレンダリングする。
macOS ネイティブのフォントレンダリングパイプラインを使うため、「OS が本来描画するはずの正解」として機能する。

### 使い方

```bash
# 等幅グリッドモード（ターミナル風、CJK 比較に推奨）
./tools/test-utils/render-text --mono --cell-info "こんにちは世界" tmp/ref.png

# SDIT の設定に合わせてフォント・サイズ・色を指定
./tools/test-utils/render-text --mono --font "Menlo" --size 14 --bg 282828 --fg EBDBB2 "テスト" tmp/ref.png
```

`--cell-info` を付けると各文字のセル境界座標が JSON で出力される。
これを使って文字単位の切り出し・右端検査が可能。

### 検討結果: 対照群の選定

| 候補 | 評価 |
|---|---|
| **CoreText 直接レンダリング（採用）** | ウィンドウ管理不要、依存なし、macOS と同じフォントパイプライン |
| ブラウザ HTML | Safari/Chrome 起動が必要、CSS のフォントメトリクスがターミナルと異なる |
| 他ターミナル比較 | インストール前提、バージョン差異あり |

CoreText が最も制御しやすく、SDIT が内部で使う cosmic-text も最終的に CoreText 経由でシェーピングするため、
比較として最も適切。

## verify-text: 3 層自動検証

`tools/test-utils/verify-text` は 3 種類のチェックを一括実行する。
エージェントは画像を読む必要がなく、テキストレポートだけで判断できる。

| チェック | 検出対象 | 依存 |
|---|---|---|
| OCR 照合 | 豆腐化、文字化け、欠落 | なし（画像のみ） |
| 輝度分析 | インク不在、右端クリッピング | `--cells`（セル境界 JSON） |
| SSIM 比較 | フォント誤り、形状歪み | `--reference`（対照群画像） |

### 右端クリッピング検出の仕組み

CJK 文字は右端に問題が起きやすい。輝度分析では:
1. セルの左半分と右半分のインク密度を比較
2. 左にインクがあるのに右にほぼないなら → クリッピングの可能性

### 使い方

```bash
render-text --mono --cell-info "テスト" tmp/ref.png | tail -n +2 > tmp/cells.json
verify-text tmp/sdit.png "テスト" --cells tmp/cells.json --reference tmp/ref.png
# → テキストレポートのみ出力。exit 0=PASS, 3=FAIL
```

### 組み込み済みテストシナリオ

- **009-cjk-display**: 3層フル検証（OCR + 輝度 + SSIM）
- **001-basic-echo**: OCR 照合（ASCII テキスト存在確認）
- **014-font-size-change**: OCR 照合（フォントサイズ変更後の描画確認）
- **020-color-emoji**: SSIM 主体（絵文字の OCR 精度が低いため）
- **023-opentype-ligature**: SSIM 主体（リガチャの OCR 認識が不安定なため）

## 設計判断の知見

### トークン効率最大化の原則

E2E テストでエージェントが画像を直接読むと画像トークンが大量に消費される。
テストツールは**テキストレポートだけを返す**設計にすることで、検証フェーズのトークンコストをほぼゼロにできる。

| アプローチ | トークンコスト |
|---|---|
| エージェントが画像を Read して視覚判定 | 非常に高い（画像トークン） |
| ツールがテキストレポートを返す | 数行のテキストのみ |

この原則は E2E テストツール全般に適用できる:
- 新しい検証ツールを作るときは、常に「エージェントが画像を読む必要があるか？」を問う
- ツール内で判定を完結させ、PASS/FAIL + 詳細をテキストで返すのが理想

**さらに根本的な問題: AI の画像 Read は品質検証に不向き**

トークンコスト以前に、AI が画像を Read しても**文字の欠損・クリッピング等の微細な問題は見落とす**。
目的を持って特定の箇所を注視しない限り、致命的な問題（画面が真っ黒、全文字が豆腐）以外は検出できない。
これはトークン効率の問題ではなく、**検出精度の問題**である。

つまり verify-text のようなツールは「コスト削減」だけでなく「品質向上」のためにも必要:
- OCR は文字単位で機械的に照合するため、人間/AI の目視より確実に欠落を検出する
- 輝度分析はピクセルレベルで右端クリッピングを検出する（目視では気づかない）
- SSIM はセル単位で数値比較するため、「なんとなく OK に見える」バイアスがない

### 画像比較アルゴリズムの落とし穴

**SSIM（構造類似度）:**
- **全体 SSIM は使えない**: ターミナル画像は背景が面積の大部分を占める。背景が同色なら文字が完全に異なっても SSIM > 0.99 になる
- **セル単位 SSIM が必須**: `--cells` でセル境界を指定し、各文字セルごとに SSIM を計算する。背景の影響を排除できる
- SSIM の閾値は 0.3 程度で十分（フォント差やアンチエイリアスの揺れがあるため高くしすぎない）

**輝度分析:**
- **「右端が背景色 = 異常」は偽陽性が多い**: 多くの文字は自然にセルの右端まで描画しない（「こ」「ち」など）
- **左右非対称検出に切り替えた**: セルの左半分と右半分のインク密度を比較し、左にだけインクがある場合のみクリッピングと判定する
- **インク有無チェック**: セル中央 60% のインク密度が背景と区別できなければ「文字が描画されていない」と判定（豆腐よりも重い欠損）

**OCR:**
- Vision.framework は CJK テキストの認識精度が高い（confidence 1.0 が普通）
- 絵文字・リガチャは認識精度が低い → SSIM で補完する設計にする
- `usesLanguageCorrection = false` で raw 認識にすると、実際の描画内容との一致度が正確になる
